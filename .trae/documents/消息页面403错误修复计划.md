# 消息页面 403 错误修复计划

## 问题描述

用户已登录，但消息页面显示"加载失败 403 需要登录才能执行"。

## 问题分析

### 根本原因

消息页面使用了 **Discourse API** 来获取通知列表（`discourse_api_service.dart`），但认证状态检查使用的是 **CoolApk API** 的认证逻辑（`auth_provider.dart` 中的 `isAuthenticatedProvider`）。

这导致：
1. 用户通过 WebView 登录 Discourse 后，Cookie 被保存到 Dio 的 CookieJar
2. 但 `isAuthenticatedProvider` 检查的是本地存储的 `uid` 和 `username`（CoolApk 登录方式）
3. 由于 Discourse 登录没有设置这些本地存储值，`isAuthenticatedProvider` 返回 `false`
4. 消息页面在 `loadNotifications()` 中检查到 `isAuthenticated` 为 `false`，显示"请先登录"
5. 同时，Discourse API 请求因为没有正确的 Cookie，返回 403 错误

## 修复方案

### 方案一：统一认证状态检查（推荐）

修改 `isAuthenticatedProvider`，使其同时支持两种登录方式的检查：

1. 检查 CoolApk 登录状态（本地存储的 uid/username）
2. 检查 Discourse 登录状态（检查 Dio CookieJar 中是否有 Discourse 的 session cookie）

**修改文件**：
- [auth_provider.dart](file:///Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/auth_provider.dart)

**具体修改**：
```dart
/// 是否已登录 Provider
///
/// 返回当前用户是否已登录（支持 CoolApk 和 Discourse 两种登录方式）
@riverpod
Future<bool> isAuthenticated(IsAuthenticatedRef ref) async {
  // 1. 检查 CoolApk 登录状态
  final user = ref.watch(currentUserProvider);
  if (user != null) return true;
  
  // 2. 检查 Discourse 登录状态（通过检查 Cookie）
  final hasDiscourseCookie = await DioClient.hasDiscourseSession();
  return hasDiscourseCookie;
}
```

同时需要在 `DioClient` 中添加 `hasDiscourseSession()` 方法：
```dart
/// 检查是否有 Discourse 登录 session
static Future<bool> hasDiscourseSession() async {
  try {
    if (_cookieJar == null) return false;
    
    final uri = Uri.parse('https://forum.trae.cn');
    final cookies = await _cookieJar!.loadForRequest(uri);
    
    // 检查是否有 Discourse 的 session cookie（通常是 _t 或 _forum_session）
    return cookies.any((cookie) => 
      cookie.name == '_t' || cookie.name == '_forum_session'
    );
  } catch (e) {
    return false;
  }
}
```

### 方案二：修改通知 Provider 的认证检查

在 `notification_provider.dart` 中，针对 Discourse 通知 API，使用专门的认证检查方式。

**优点**：改动范围小，不影响其他功能
**缺点**：代码重复，不够优雅

### 方案三：WebView 登录后同步用户信息

在 WebView 登录 Discourse 成功后，从 Discourse 获取用户信息并保存到本地存储。

**优点**：完全统一登录状态
**缺点**：需要额外获取 Discourse 用户信息，改动较大

## 推荐方案：方案一

选择方案一，因为：
1. 改动范围适中，只修改认证相关的 Provider
2. 能够正确反映用户的实际登录状态
3. 不影响现有 CoolApk 登录逻辑
4. 为后续支持双登录方式打下基础

## 实施步骤

1. **修改 DioClient**：添加 `hasDiscourseSession()` 方法
2. **修改 auth_provider.dart**：更新 `isAuthenticatedProvider` 为异步 Provider，支持双登录方式检查
3. **修改 notification_provider.dart**：更新 `loadNotifications()` 等方法的认证检查调用方式
4. **修改 message_provider.dart**（如需要）：同样更新认证检查
5. **测试验证**：
   - WebView 登录 Discourse 后，检查消息页面是否正常加载
   - 未登录时，检查是否正确显示登录提示
   - CoolApk 登录（如有）是否正常工作

## 注意事项

1. `isAuthenticatedProvider` 从同步 Provider 改为异步 Provider 后，所有使用它的组件需要处理 `AsyncValue` 状态
2. 需要确保 `DioClient` 的 CookieJar 已经初始化后再检查 Discourse session
3. 403 错误处理：如果 API 返回 403，应该提示用户重新登录，并清除过期的 Cookie
