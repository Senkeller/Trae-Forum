## 4. 关键发现（按优先级）

## P0（必须先处理）

### P0-1 关注/取关分支存在逻辑错误（会导致错误调用）

**Evidence**

- 使用 `url.contains('follow')` 判断操作类型：  
  `'/v6/user/unfollow'` 也会命中 `follow`。  
  见 [api_service.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/core/network/api_service.dart:830)
- 误调用 follow 分支：  
  [api_service.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/core/network/api_service.dart:832)
- 取关入口传值：  
  [user_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/user_provider.dart:535)

**Impact**

1. 用户点击“取关”可能仍发“关注”请求。
2. UI 状态和服务端状态容易出现反向漂移。

**建议**

1. 用明确枚举/布尔参数表达动作（`action=follow|unfollow`），禁止字符串模糊匹配。
2. 增加单元测试覆盖 follow/unfollow 两分支。

**Evidence**

- 缺失依赖文件引用：  
  [composer_editor.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/widgets/editor/composer_editor.dart:4)
- 明显字符串错误（`$before1` 未定义）：

**建议**

### P0-3 登录门禁未收敛到路由层，导致行为不一致

**Evidence**

- 路由守卫仍是 TODO：  
  [routes.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/config/routes.dart:323)
- 同时存在同步与异步登录判定混用：
  - 同步：  
    [feed_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/feed_provider.dart:346)
  - 异步：  
    [message_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/message_provider.dart:263)  
    [notification_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/notification_provider.dart:119)

**Impact**

1. 容易出现“页面显示已登录，但某些操作仍提示未登录”。
2. 登录态判定时机不统一，触发偶发态问题。

**建议**

1. 路由层加统一门禁（白名单路由+受保护路由）。
2. 业务层统一使用单一登录状态源（推荐异步可恢复态）。

## P1（高优先级，紧随 P0）

### P1-1 Dashboard 状态管理存在重复实现

**Evidence**

- 同一文件存在两套状态体系：
  - `TraeDashboardNotifier`（StateNotifier）
  - `DashboardStateNotifier`（@riverpod AsyncValue）
  见 [trae_dashboard_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/trae_dashboard_provider.dart:47), [trae_dashboard_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/trae_dashboard_provider.dart:124)
- `traeDashboardNotifierProvider` 未发现调用路径，疑似遗留：  
  [trae_dashboard_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/trae_dashboard_provider.dart:117)

**Impact**

1. 后续维护时容易在错误的状态树上改动。
2. 调试与问题定位成本上升。

**建议**

1. 保留一套状态模型，另一套删除或迁移。
2. 明确“页面级 Provider”与“聚合 Provider”边界。

**Evidence**

- `search_provider_new.dart` 存在但未发现引用：  
  [search_provider_new.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/search_provider_new.dart:1)

2. 将 `/search/result` 接回实际数据源或下线该路由。

### P1-3 网络层存在语义遗留与头信息不正确风险

**Evidence**

- 业务层仍传 `/v6/...` 语义参数，但实现已映射到 Discourse：  
  [feed_provider.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/presentation/providers/feed_provider.dart:356)
- `AuthInterceptor` 将 `StorageKeys.deviceId` 常量名直接当 Header 值：  
  [auth_interceptor.dart](/Users/jason/Documents/codex/TraeU/traeu/lib/core/network/interceptors/auth_interceptor.dart:35)

**Impact**

1. 后续开发者对 API 语义理解偏差。
2. 设备标识 Header 不可信，影响服务端分析能力。

**建议**

1. 去语义债：参数命名与实际请求行为一致。
2. 设备 ID 从真实存储读取，避免发送键名字面量。

## 6. 风险与应对

1. **风险：** 清理重复状态时引入行为回归。  
   **应对：** 先补回归测试再删旧实现。

2. **风险：** 登录链路修正后暴露更多历史脏数据。  
   **应对：** 保留日志分层与错误码透传，避免“网络异常”掩盖真实原因。

3. **风险：** 测试大量失效导致短期交付压力。  
   **应对：** 先修关键链路用例，分批恢复非关键 UI 断言。
