## 一、项目概述

| 项目 | 技术栈 | 数据源 | 目标平台 | 代码量 |
|------|--------|--------|----------|--------|
| **c001apk** | Android (Kotlin) + Retrofit + Hilt + Room | 酷安网 API (非官方逆向) | Android | ~150+ Kotlin 文件 |
| **trae-forum** | Flutter + Riverpod + Dio + GoRouter | TRAE 论坛 Discourse API | 跨平台 (iOS/Android/Web) | ~100+ Dart 文件 |

### 1.1 c001apk 项目简介
c001apk 是一个针对酷安网的非官方 Android 客户端，使用 Kotlin 开发，通过逆向酷安网移动端 API 实现完整功能。

**核心技术**:
- Retrofit 2.9.0 + OkHttp 4.12.0 网络请求
- Hilt 2.48 依赖注入
- Room 2.6.1 本地数据库
- Glide 4.16.0 图片加载
- Kotlin Coroutines + Flow 异步处理

### 1.2 trae-forum 项目简介
trae-forum 是使用 Flutter 重写的 TRAE 论坛客户端，目标是通过 Flutter 实现跨平台兼容。

**核心技术**:
- Flutter 3.x + Riverpod 状态管理
- Dio 网络请求
- GoRouter 路由管理
- 适配 Discourse 官方 API

## 二、架构设计对比

### 2.1 整体架构

| 维度 | c001apk | trae-forum |
|------|---------|------------|
| **架构模式** | MVVM + Clean Architecture | Riverpod + 适配器模式 |
| **依赖注入** | Hilt | Riverpod Provider |
| **网络层** | Retrofit + OkHttp | Dio |
| **本地存储** | Room + DataStore | SharedPreferences (部分) |
| **图片加载** | Glide | cached_network_image |
| **路由管理** | Intent + ActivityResultLauncher | GoRouter |

### 2.2 架构设计优缺点分析

#### 2.2.1 c001apk 架构优缺点

**优点**:

| 方面 | 说明 |
|------|------|
| **Clean Architecture 分层清晰** | `logic/` 层独立于 UI，`repository` 模式解耦数据源，利于单元测试和后期维护 |
| **Hilt 依赖注入规范** | 全局单例管理，生命周期与 Activity/Fragment 绑定，内存泄漏风险低 |
| **Room 数据库完善** | 类型安全、编译时校验、Flow 响应式查询，适合复杂本地数据场景 |
| **Retrofit 接口类型安全** | 注解定义 API，编译期检查参数，IDE 自动补全支持好 |
| **ViewModel + LiveData** | 生命周期感知，配置变更时数据自动恢复 |
| **协程 + Flow** | 简洁的异步处理，Flow 适合响应式数据流 |

**缺点**:

| 方面 | 说明 |
|------|------|
| **目录结构过深** | `ui/activity/` + `ui/fragment/` + `ui/viewmodel/` 分离，同一功能代码分散 |
| **Hilt 过度设计** | 对于中小型项目，Hilt 模板代码量大，学习成本高 |
| **Activity/Fragment 臃肿** | 业务逻辑虽在 ViewModel，但 UI 绑定代码仍较重 |
| **Glide 内存占用** | 功能强大但包体积大，对于图片场景简单的 App 略显笨重 |
| **非响应式数据流** | LiveData 更适合 UI 状态，Repository 层仍用 suspend fun，响应式链不完整 |
| **缺乏真正的 Compose 支持** | 仍使用传统 View 系统，未能发挥 Kotlin 协程全部潜力 |

#### 2.2.2 trae-forum 架构优缺点

**优点**:

| 方面 | 说明 |
|------|------|
| **Riverpod 轻量灵活** | 无需继承、无 Context 依赖，编译时生成代码，利于 tree-shaking |
| **Flutter 跨平台** | 一套代码支持 iOS/Android/Web，UI 一致性好 |
| **适配器模式清晰** | `data/adapters/` 分离数据转换逻辑，UI 层只关心业务模型 |
| **GoRouter 声明式路由** | 类型安全、深度链接支持、路由守卫机制完善 |
| **Dio 拦截器强大** | 统一错误处理、日志打印、Token 刷新逻辑集中 |
| **Flutter Hot Reload** | 开发效率高，UI 修改即时预览 |

**缺点**:

| 方面 | 说明 |
|------|------|
| **SharedPreferences 能力有限** | 只适合存储简单 KV 数据，无法处理复杂结构化数据 |
| **缺少离线支持** | 无本地数据库，缓存策略弱，离线体验差 |
| **Dio 响应式不足** | 返回 Future 而非 Stream，长轮询/实时更新需额外处理 |
| **图片加载生态弱** | `cached_network_image` 功能单一，无 Glide 的占位图/过渡动画丰富 |
| **平台通道复杂性** | 原生功能扩展需要 Platform Channel，增加维护成本 |
| **Flutter 包体积** | release 包比原生 Android 大 3-5MB (压缩后) |
| **内存占用较高** | Flutter Engine + Dart VM 运行时不比轻量原生 App 低 |

#### 2.2.3 架构选择建议

| 场景 | 推荐方案 |
|------|----------|
| **数据复杂、离线优先** | 借鉴 c001apk 的 Room 数据库方案 |
| **跨平台优先、快速迭代** | 保持 trae-forum 的 Flutter 架构 |
| **需要复杂本地存储** | 为 trae-forum 添加 Isar/Hive 等本地数据库 |
| **实时性要求高** | 为 trae-forum 添加 WebSocket 或 SSE 支持 |
